# 위협모형

| 항목 | 내용 |
|---|---|
| 대상 시스템 | AI 모델 평가 시스템 / XAI 분석 시스템 / AI 보안 논문 작성 프로세스 |
| 보호 자산 | 평가 데이터, hidden test, benchmark metadata, 모델 버전, prompt/template, 실험 로그, 설명 결과, 참고문헌, AI 활용기록 |
| 공격자 | 평가 데이터 오염자, 모델 개발자, 논문 작성자, AI 도구 오용자, 설명 결과를 악용하는 외부 관찰자 |
| 공격자의 지식 | White-box, gray-box, black-box 모두 가능. 논문 작성 프로세스에서는 내부 작성자가 가장 강한 지식을 가진다. |
| 공격자의 능력 | 평가 데이터 노출, benchmark 오염, 반복 질의, 실험결과 과장, 참고문헌 조작, AI 산출물 무검증 사용, 설명 결과를 통한 민감 단서 추론 |
| 공격 경로 | 평가 데이터셋, 실험 환경, 설명가능성 도구, 참고문헌 목록, AI 활용기록, 제출용 보고서 |
| 공격 성공 조건 | 성능이 과대평가되거나, 검증 불가능한 결과가 제출되거나, 설명 결과가 민감정보 또는 모델 우회 단서를 노출함 |
| 방어자 가정 | DOI/URL 확인, 로컬 PDF 대조, 실행 로그 저장, AI 활용 고지, 재현성 점검, 대체 PDF 표시가 가능하다. |
| 제외 범위 | 실제 학술 부정행위 수행, 허위 결과 조작, 실제 심사 시스템 공격, 실제 개인정보 사용, 실제 상용 API 무단 공격 |

## 연구문제 후보

RQ1. LLM과 AI 보안 연구에서 benchmark contamination과 evaluation leakage는 평가 신뢰성에 어떤 영향을 미치는가?

RQ2. XAI 기반 설명은 보안 평가에서 신뢰성 증거로 사용할 수 있는가, 아니면 새로운 공격면으로 보아야 하는가?

RQ3. AI 보안 논문에서 재현성, 참고문헌 검증, AI 활용 고지를 포함한 연구윤리 체크리스트는 어떻게 구성해야 하는가?

## 위협 단계별 정리

| 단계 | 실패 조건 | 최소 통제 |
|---|---|---|
| 평가 데이터 선정 | 평가셋이 학습/튜닝/프롬프트 설계에 노출됨 | benchmark provenance, 중복 검사 |
| 실험 실행 | seed, config, 코드 버전, 로그가 남지 않음 | Dockerfile, config.yaml, outputs 보존 |
| XAI 분석 | 설명이 민감 feature 또는 모델 우회 단서를 노출함 | 공개 범위 제한, stability/leakage 점검 |
| 논문 작성 | DOI, 저자, 출판연도, 대체 PDF 상태를 검증하지 않음 | reference verification table |
| AI 활용 | AI 산출물을 검증 없이 본문에 반영하거나 사용을 숨김 | AI disclosure, human review |
