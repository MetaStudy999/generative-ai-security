W07 LLM 학습·정렬·평가 & 보안·프라이버시

LLM 보안 평가는 utility, ASR, leakage, refusal, code risk, 재현성을 함께 기록해야 한다.

1. 왜 중요한가

  • LLM은 prompt, context, output, log, benchmark가 얽힌 시스템이다.
  • 평가가 오염되면 성능 해석과 보안 판단이 함께 흔들린다.

2. AI 원리 70%

개념보안 연결
Pretrainingmemorization, data extraction
Alignment/RLHFrefusal quality
Context windowprompt injection/leakage
Benchmarkcontamination

3. 보안 이슈 30%

위협대표 지표
Data extractionprivacy leakage
Prompt injectionASR
Insecure codevulnerability rate
Over-refusalanswer rate

4. 논문 5편의 역할

P01은 평가, P02/P03은 보안·프라이버시, P04는 MLLM, P05는 소프트웨어 보안 접점을 담당한다.

5. 위협모형

User prompt -> Context window -> LLM response -> Logs / Code / Benchmark

실제 개인정보 추출, 실제 서비스 공격, exploit instruction은 제외한다.

6. 평가 프로토콜

항목지표

Attack successASR
Privacyleakage rate
Refusalrefusal quality, over-refusal
Code securityvulnerability rate

7. Synthetic Toy 실험

  • 조건: clean, prompt attack simulation, privacy-risk, code security
  • 각 조건 40개 synthetic sample
  • Guard threshold 0.55
  • 실제 LLM/API 호출 없음

9. 해석

  • ASR만 보면 utility와 over-refusal을 놓친다.
  • Utility만 보면 leakage와 code risk를 놓친다.
  • Code security 조건은 과차단과 취약 코드 위험을 함께 봐야 한다.

10. 기말논문 연결

LLM/RAG 기반 AI 시스템의 보안·프라이버시·재현성 평가 프레임워크.

관련연구, 위협모형, 평가방법, 재현성 장에 반영.

11. 결론

  • LLM 보안 평가는 다중지표 문제다.
  • 수치는 seed, config, CSV/JSON/run log가 있을 때만 주장한다.
  • W07은 기말논문의 평가 프레임워크로 직접 연결된다.
