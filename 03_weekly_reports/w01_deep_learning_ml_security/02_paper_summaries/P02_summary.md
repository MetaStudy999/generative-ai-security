## 2. 한 문장 요약

> 이 논문은 ML 시스템을 데이터 관리, 모델 학습, 모델 검증, 배포 단계의 생명주기로 나누고, 각 단계에서 안전성과 신뢰성을 입증하기 위한 보증 요건과 방법을 정리한다.

## 3. 연구문제

핵심 질문은 “안전 중요 시스템에 ML을 넣을 때 어떤 증거가 있어야 충분히 안전하다고 말할 수 있는가”이다. W01에서는 이 질문을 보안 관점으로 확장해 데이터 오염, 평가셋 편향, 배포 후 drift, 로그 누락 같은 위험을 생명주기 단계별로 배치한다.

## 4. 핵심 개념

| 개념 | 설명 | 보안 연구 연결 |
|---|---|---|
| ML lifecycle | 데이터 수집부터 배포와 운영까지 반복되는 ML 개발 흐름이다. | 공격면을 단일 모델이 아니라 전체 파이프라인으로 확장한다. |
| Assurance evidence | 시스템이 요구사항을 만족한다는 주장에 필요한 검증 증거이다. | 실험 로그, 설정, 데이터 버전, threat model이 보증 증거가 된다. |
| Desiderata | 각 단계에서 달성해야 하는 바람직한 성질이다. | 정확도뿐 아니라 robustness, privacy, traceability가 요구된다. |
| Open challenges | 기존 방법으로 충분히 해결되지 않은 보증 난점이다. | 분포 변화, 설명가능성, 자동화된 검증 한계와 연결된다. |

## 5. 방법론

문헌조사 기반으로 ML 생명주기를 단계화하고, 단계별 입력/출력, 활동, 보증 목표, 사용 가능한 방법을 정리한다. 본 보고서에서는 이 구조를 W01 Research Track의 위협모형과 평가 프로토콜 틀로 사용했다.

## 6. 주요 결과

ML 보증은 모델 학습 이후 한 번 수행하는 테스트가 아니라, 데이터 관리부터 배포 이후 모니터링까지 이어지는 증거 축적 과정이다. 각 단계는 서로 의존하므로 학습 데이터 품질, 모델 검증, 운영 환경 가정이 함께 기록되지 않으면 최종 성능 수치만으로 신뢰성을 설명할 수 없다.

## 7. 보안 관점 분석

P02는 보안 위협을 생명주기별로 위치시키는 데 가장 직접적으로 활용된다. 데이터 수집 단계에서는 poisoning과 privacy leakage, 모델 학습 단계에서는 backdoor와 overfitting, 검증 단계에서는 평가셋 오염과 지표 왜곡, 배포 단계에서는 evasion과 model extraction 위험을 다룰 수 있다.

## 8. 한계와 오픈문제

보증 프레임은 넓지만 특정 공격별 수학적 평가식이나 실험 프로토콜은 별도 설계가 필요하다. 또한 현실 시스템에서는 데이터·모델·배포 로그가 조직 경계에 흩어져 있어 보증 증거의 완전성을 확보하기 어렵다.

## 9. 기말 논문에 반영할 부분

기말 논문에서는 P02의 생명주기 틀을 중심 구조로 삼아 W01 이후 주차의 공격 유형을 단계별로 매핑한다. 특히 “보안성 평가 = 성능 테스트 + 위협모형 + 재현성 증거”라는 논지의 핵심 근거로 반영한다.
